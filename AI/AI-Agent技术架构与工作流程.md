## 一句话理解

**AI Agent = 能自主"规划 + 决策 + 调用工具 + 执行"的 AI 工作者**，不像ChatGPT只能回答问题，Agent能帮你完成具体任务。

## Planning 阶段（规划）

### 1. RAG - 知识检索增强

```
┌─────────────────────────────────────────┐
│              RAG 工作流程                 │
└─────────────────────────────────────────┘

用户问题: "帮我看看这个合同的潜在风险"
       │
       ▼
┌──────────────────┐
│   向量化 (Embedding) │ ← 把文本转为向量
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│   向量数据库       │ ← Milvus / Pinecone / Chroma
│   相似度搜索      │
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│   Top-K 文档      │ ← 找到最相关的K个文档
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│  构造 Prompt      │ ← 把相关文档拼接进提示词
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│   LLM 生成回答    │
└──────────────────┘
```

**解决的问题**：模型"不知道你私有数据" + 模型"容易胡说八道"

### 2. Prompt 工程

```
┌─────────────────────────────────────────┐
│           Prompt 常见技巧                 │
└─────────────────────────────────────────┘

1. Few-shot (少样本提示)
   ┌────────────────────────────┐
   │ 例：                      │
   │ 输入：2+2=？             │
   │ 输出：4                   │
   │                           │
   │ 输入：3+3=？             │
   │ 输出：？                  │
   └────────────────────────────┘

2. Chain-of-Thought (思维链)
   ┌────────────────────────────┐
   │ 例：                      │
   │ 问题：小明有5个苹果，     │
   │      吃了2个，又买了3个， │
   │      现在有几个？         │
   │                           │
   │ 思考：5-2+3=6            │
   │ 答案：6个                 │
   └────────────────────────────┘

3. Role-playing (角色扮演)
   ┌────────────────────────────┐
   │ 例：                      │
   │ "你是一位资深律师，       │
   │   请帮我审查这份合同..."  │
   └────────────────────────────┘

4. Tree-of-Thought (思维树)
   ┌────────────────────────────┐
   │        问题               │
   │       /  │  \            │
   │    方案A 方案B 方案C      │
   │     /|\   /|\   /|\       │
   │   ...  ...  ...          │
   │                           │
   │   评估所有分支，选择最优  │
   └────────────────────────────┘
```

### 3. CoT / ToT 思维链

```
┌─────────────────────────────────────────┐
│         CoT vs ToT 对比                 │
└─────────────────────────────────────────┘

CoT (Chain-of-Thought 思维链):
┌────────────┐    ┌────────────┐    ┌────────────┐
│   问题    │ → │ 思考过程  │ → │   答案    │
└────────────┘    └────────────┘    └────────────┘
                   (线性步骤)          

ToT (Tree-of-Thought 思维树):
        ┌────────────┐
        │   问题    │
       / \         / \
   ┌──────┐   ┌──────┐   ┌──────┐
   │方案A │   │方案B │   │方案C │
   │      │   │      │   │      │
   └──────┘   └──────┘   └──────┘
       \        |        /
        \   ┌───┴───┐   /
         \ /         \ /
        ┌──────────────┐
        │   评估选择   │
        │   最优方案   │
        └──────────────┘
        (树状搜索)
```

## Execution 阶段（执行）

### 1. Code Interpreter - 代码执行器

```
┌─────────────────────────────────────────┐
│         Code Interpreter 工作流程          │
└─────────────────────────────────────────┘

1. LLM 生成代码
   ┌────────────────────────────┐
   │  "计算 1-100 的素数和"   │
   └────────────┬─────────────┘
                │
                ▼
2. 发送到沙箱环境执行
   ┌────────────────────────────┐
   │  Docker 容器 / Python 进程 │
   └────────────┬─────────────┘
                │
                ▼
3. 执行并返回结果
   ┌────────────────────────────┐
   │  1060                     │
   └────────────┬─────────────┘
                │
                ▼
4. LLM 解读结果
   ┌────────────────────────────┐
   │  "1-100的素数和是1060"    │
   └────────────────────────────┘

典型应用：
- GPT-4 Code Interpreter
- Claude Code Runner
- 数学计算
- 数据分析
- 文件处理
```

### 2. Tool Use - 工具调用

```
┌─────────────────────────────────────────┐
│           Tool Use 工作流程              │
└─────────────────────────────────────────┘

Step 1: 定义工具
┌────────────────────────────────────┐
│  tools: [                           │
│    {                                │
│      "name": "weather",            │
│      "description": "查询天气",    │
│      "parameters": {               │
│        "type": "object",          │
│        "properties": {            │
│          "city": {"type": "string"}│
│        }                           │
│      }                             │
│    }                               │
│  ]                                  │
└────────────────────────────────────┘
                │
                ▼
Step 2: LLM 判断调用哪个工具
   "北京天气怎么样？" → 需要调用 weather(city="北京")
                │
                ▼
Step 3: 执行工具返回结果
   { "city": "北京", "temp": "15℃", "weather": "晴" }
                │
                ▼
Step 4: LLM 生成最终回答
   "北京今天天气晴，气温15℃。"

常见工具：
- 搜索 (Web Search)
- 文件读写 (File I/O)
- API 调用 (HTTP Request)
- 数据库查询 (Database)
- 浏览器控制 (Browser)
```

### 3. MCP - Model Context Protocol

```
┌─────────────────────────────────────────┐
│        MCP (Model Context Protocol)      │
│        = AI 世界的 USB 接口              │
└─────────────────────────────────────────┘

传统方式 (每个AI需要单独适配):
   ┌─────────┐     ┌─────────┐     ┌─────────┐
   │  ChatGPT │ ──→ │ Notion  │     │  Slack  │
   │   LLM    │     │   API   │     │   API   │
   └─────────┘     └─────────┘     └─────────┘
   每个AI都要写一个 Notion 适配器
   每个AI都要写一个 Slack 适配器
          ↓
MCP 方式 (统一协议):
   ┌─────────┐     ┌─────────────────┐
   │ 任意AI  │ ──→ │     MCP Server  │ ←→ Notion
   │  LLM    │     │   (统一接口)   │ ←→ Slack
   └─────────┘     │                 │ ←→ FileSystem
                   └─────────────────┘
                   (即插即用)

┌─────────────────────────────────────────┐
│           MCP 架构                       │
└─────────────────────────────────────────┘

┌─────────┐     ┌────────────┐     ┌─────────────────┐
│  Host   │ ←→ │   Client   │ ←→ │     Server      │
│ (LLM)   │    │  (MCP SDK) │    │ (各种服务)      │
└─────────┘     └────────────┘     └─────────────────┘
                            │
                            ▼
              ┌─────────────────────────┐
              │    Resources 资源        │ ← 文件、数据库
              │    Tools 工具           │ ← API、函数
              │    Prompts 提示词        │ ← 模板
              └─────────────────────────┘

MCP 生态:
- 文件系统 (File System)
- GitHub / Git
- Notion
- Slack
- PostgreSQL
- Puppeteer (浏览器)
- 等等...
```

## Multi-Agent 协作模式（多智能体）

### 1. Supervisor 模式（监督者）

```
┌─────────────────────────────────────────┐
│        Supervisor (监督者模式)           │
└─────────────────────────────────────────┘

                    ┌─────────────┐
                    │  Supervisor │
                    │   (管理者)   │
                    └──────┬──────┘
                           │
         ┌───────────────┼───────────────┐
         │               │               │
         ▼               ▼               ▼
    ┌─────────┐   ┌─────────┐   ┌─────────┐
    │ Agent A  │   │ Agent B  │   │ Agent C │
    │ (搜索)   │   │ (分析)   │   │ (整理)  │
    └─────────┘   └─────────┘   └─────────┘
         │               │               │
         └───────────────┼───────────────┘
                         │
                         ▼
                    ┌─────────────┐
                    │  Supervisor │
                    │  汇总结果   │
                    └─────────────┘

适用场景：
- 需要根据任务类型分配给不同Agent
- 需要一个"管理者"协调
```

### 2. Sequential 模式（顺序执行）

```
┌─────────────────────────────────────────┐
│       Sequential (顺序执行模式)          │
└─────────────────────────────────────────┘

┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ Agent 1 │ →  │ Agent 2 │ →  │ Agent 3 │ →  │  输出   │
│ 数据收集 │    │ 数据处理│    │ 结果整理 │    │        │
└─────────┘    └─────────┘    └─────────┘    └─────────┘
   输出1          输出2          输出3
                            │
                            └────────────→ 最终结果

适用场景：
- 流水线式任务
- 每一步依赖上一步输出
- 数据处理流程
```

### 3. Network 模式（网状协作）

```
┌─────────────────────────────────────────┐
│         Network (网状协作模式)           │
└─────────────────────────────────────────┘

    ┌─────────┐
    │ Agent A │
    └────┬────┘
         │
    ┌────┼────┐
    │         │
    ▼         ▼
┌───────┐ ┌───────┐
│Agent B│ │Agent C│
└───┬───┘ └───┬───┘
    │         │
    └────┬────┘
         │
         ▼
    ┌─────────┐
    │ Agent D │ ← 结果汇总
    └─────────┘

特点：
- 去中心化
- 任意Agent可以相互通信
- 适合复杂协作场景
```

### 4. Hub-and-Spoke 模式（中心辐射）

```
┌─────────────────────────────────────────┐
│    Hub-and-Spoke (中心辐射模式)          │
└─────────────────────────────────────────┘

                ┌─────────────┐
                │     Hub     │
                │  (中央)    │
                └──────┬──────┘
                       │
       ┌───────────────┼───────────────┐
       │               │               │
       ▼               ▼               ▼
   ┌─────────┐   ┌─────────┐   ┌─────────┐
   │  Spoke  │   │  Spoke  │   │  Spoke  │
   │   A     │   │   B     │   │   C     │
   └─────────┘   └─────────┘   └─────────┘

适用场景：
- 中央Agent作为协调中心
- 辐射出多个专业Agent
- 需要统一入口
```

## Agent vs Chatbot 对比

| 特性 | Chatbot | Agent |
|------|---------|-------|
| **能力** | 回答问题 | 完成任务 |
| **交互** | 被动响应 | 主动执行 |
| **工具** | 无 | 有 (Tool Use) |
| **规划** | 无 | 有 (Planning) |
| **记忆** | 短期 | 长期 + 向量存储 |
| **自主性** | 低 | 高 |
| **例子** | ChatGPT | AutoGPT, Claude Agent |

## 面试常见问题

### Q1: Agent 和传统程序有什么区别？

```
┌─────────────────────────────────────────┐
│  传统程序 vs Agent                      │
├─────────────────────────────────────────┤
│                                         │
│  传统程序:                               │
│  输入 → 固定逻辑 → 输出                  │
│  (确定性执行)                            │
│                                         │
│  Agent:                                 │
│  输入 → LLM推理 → 规划 → 工具调用 → 输出 │
│  (动态决策)                              │
│                                         │
│  核心区别: "确定性" vs "智能性"          │
└─────────────────────────────────────────┘
```

### Q2: 如何保证 Agent 的可靠性？

```
┌─────────────────────────────────────────┐
│        Agent 可靠性保证措施              │
└─────────────────────────────────────────┘

1. 错误处理
   - 工具调用失败重试
   - 降级策略
   - 异常状态捕获

2. 验证机制
   - 输出格式验证
   - 结果检查点
   - 人工审核入口

3. 监控日志
   - 完整执行轨迹
   - 关键节点记录
   - 性能监控

4. 安全防护
   - 敏感操作确认
   - 权限控制
   - 审计日志
```
