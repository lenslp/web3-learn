## 一、本地模型：Ollama

Ollama 是在本地运行大语言模型的工具，无需联网，数据不外传。

常用命令：

```
ollama run qwen3:8b # 运行模型
ollama list # 查看已安装模型
ollama pull bge-m3 # 下载 Embedding 模型
```

模型选择建议：

| 模型 | 适用场景 | 显存需求 |
|------|---------|---------|
| qwen3-vl:8b | 图像理解（视觉模型） | ~8GB |
| qwen3:8b | 纯文本对话 / RAG | ~8GB |
| qwen3:14b | 效果更好的文本任务 | ~16GB |

## 二、Embedding 与 RAG

### 什么是 Embedding

将文字转换成一串数字（向量），让计算机能理解语义相似度。

- "猫" 和 "猫咪" → 向量非常接近
- "猫" 和 "汽车" → 向量相差很远

### 为什么用 bge-m3

- 专为中英文优化，中文语义匹配效果好
- 支持长文本（最长 8192 token）
- 同时支持 dense / sparse / multi-vector 三种检索方式
- 完全免费，可本地运行

### RAG 常见问题：回答跑偏

**原因：** 引用内容过多，小模型注意力被大量上下文淹没，忘记了用户的真实问题。

**解决方法：**
- 引用数量从 18 减到 3～5 条
- System Prompt 中加入：「如果参考材料与问题无关，忽略材料，用自身知识回答」

### 为什么要分片（Chunking）

**核心原因：大模型有上下文限制**

假设你有 100 万份文档，你不可能把它们全塞给大模型，因为：
- context window 有限（放不下）
- 即使放得下，推理成本极高

所以需要先快速找到相关内容，再把相关内容交给大模型处理。

### Embedding 做的事

把文字转成一串数字（向量），语义相近的内容在向量空间里距离也近：

```
"苹果手机" → [0.2, 0.8, 0.1, ...]
"iPhone"   → [0.21, 0.79, 0.11, ...]  # 很近
"香蕉"     → [0.9, 0.1, 0.6, ...]      # 很远
```

这样就能用数学运算（余弦相似度）快速在海量数据里找到语义最相关的内容，速度远快于让大模型逐一阅读。

### 整个 RAG 流程

```
用户提问
   ↓
embedding 向量化提问
   ↓
在向量数据库里快速找到相关文档（毫秒级）
   ↓
把相关文档 + 问题一起交给大模型
   ↓
大模型生成回答
```

**简单说：embedding 负责"找"，大模型负责"理解和回答"，两者分工协作。

## 三、Modelfile

Ollama 专属的模型配置文件，类似 Docker 的 Dockerfile，用于固化模型行为。

### 基本结构

```dockerfile
FROM qwen3:8b

SYSTEM """
你是一个专业的技术助手。
如果参考材料与问题无关，忽略材料，用自身知识回答。
"""

PARAMETER temperature 0.7
PARAMETER num_ctx 4096
```

### 使用步骤

```bash
touch Modelfile # 创建文件
ollama create 我的助手 -f Modelfile # 构建模型
ollama run 我的助手 # 运行模型
```

### 核心参数

| 参数 | 作用 | 推荐值 |
|------|------|--------|
| temperature | 越低越稳定，越高越有创意 | 0.5～0.7 |
| num_ctx | 上下文长度，影响记忆能力 | 4096～8192 |
| top_p | 控制词语选择范围 | 0.9 |

### Cherry Studio 与 Modelfile 的关系

Cherry Studio 把 Modelfile 的参数可视化了，本质是同一回事。

> ⚠️ 注意默认设置中「上下文数」只有 16，RAG 场景应调至 4096。

## 四、模型微调

### 什么是模型权重

模型权重 = 模型的「记忆」，是模型文件（.gguf）里存储的数十亿个小数。

微调的本质是**修改这些数字的值**，而不是增加数字的数量，因此：
- 微调前后模型大小基本不变
- Modelfile 只改变「使用方式」，微调才改变「大脑本身」

### 主要微调方法

| 方法 | 原理 | 显存需求 | 适合人群 |
|------|------|---------|---------|
| 全量微调 | 更新所有参数 | 80GB+ | 大型团队 |
| LoRA | 只训练少量附加参数 | ~16GB | 个人 / 小团队 |
| QLoRA | LoRA 的量化版 | ~8GB | 消费级 GPU / Mac |

## 五、LoRA 详解

> 核心公式：基座模型（插座）+ Adapter（插头）= 新能力

### 什么是 Adapter

LoRA 训练时不修改原模型，而是在旁边训练一个「补丁文件」：

- 原始权重冻结不动
- Adapter 只有几十 MB（vs 基座模型的几 GB）
- 推理时两者叠加，效果等于微调过的模型
- 可以随时插拔，换不同 Adapter 得到不同能力

### 训练数据格式

```json
{
 "instruction": "Embedding 是什么？",
 "input": "",
 "output": "Embedding 是将文字转换成向量的技术..."
}
```

> 数据质量 >> 数据数量，100 条精心写的数据胜过 1000 条粗糙数据。

### 推荐训练框架：LLaMA-Factory

```bash
git clone https://github.com/hiyouga/LLaMA-Factory
pip install -e ".[torch,metrics]"
llamafactory-cli webui # 启动可视化界面
```

### 完整流程

```
准备数据 → 选择基座模型 → 配置训练参数 → 开始训练 → 合并 Adapter → 评估效果
```

### 硬件建议

| 硬件 | 能否训练 | 说明 |
|------|---------|------|
| Mac / 仅 CPU | ⚠️ 勉强 | QLoRA 可跑，但速度极慢 |
| RTX 3090 / 4090 | ✅ | LoRA 的最低可用门槛 |
| 云 GPU（AutoDL） | ✅ 推荐 | 几块钱 / 小时，适合学习 |
